## 1. Постановка задачи

Команда ecom.tech автоматизирует **reverse engineering документации**: нужно извлекать описания REST API-эндпоинтов и Kafka-продюсеров/консьюмеров непосредственно из исходного кода — без ручного труда. Масштаб: **сотни микросервисов** на пяти языках — Golang, Kotlin, Ruby, Elixir, PHP.

Задача не тривиальная, потому что у каждого языка своя экосистема фреймворков и свои архитектурные соглашения о том, где живут эндпоинты:

| Язык | REST-эндпоинты | Kafka-продюсеры | Kafka-консьюмеры |
|---|---|---|---|
| **Golang** | `http.HandleFunc`, Gin/Echo `.GET`/`.POST` | `producer.SendMessage` (sarama) | `consumer.Subscribe` (sarama) |
| **Kotlin** | `@RestController`, `@RequestMapping`, Ktor `routing { get("/") { } }` | `KafkaTemplate.send` | `@KafkaListener` (Spring) |
| **Ruby** | Rails-контроллеры + `routes.rb` (`resources`, `get`) | `producer.produce` (ruby-kafka) | `consumer.subscribe` |
| **Elixir** | `Phoenix.Router` + макросы `get "/", PageController, :index` | `KafkaEx.produce` | `GenConsumer` |
| **PHP** | Laravel `routes/api.php` (callback или `[Controller::class, 'method']`), Symfony-аннотации | `KafkaProducer::send` (php-rdkafka) | consumer-groups |

Цель инструмента: точно и воспроизводимо находить все эти паттерны в незнакомой кодовой базе, не требуя ручной настройки под каждый проект.

## 2. Эволюция подходов: три тупика

### Подход 1 — Tree-sitter напрямую

Tree-sitter строит полноценное AST и понимает синтаксис — казалось бы, идеально. Проблема: **AST видит синтаксис, но не семантику фреймворка**.

Вызов `router.GET("/orders", handler)` синтаксически неотличим от любого другого вызова метода на объекте. Если в проекте используются:
- обёртки над HTTP-библиотеками (базовые контроллеры),
- DSL (кастомные роутеры),
- метапрограммирование (Ruby `define_method`, Elixir-макросы),

...то tree-sitter честно строит AST, но правила поверх него либо пропускают реальные эндпоинты, либо дают ложные срабатывания. Плюс: для каждой пары язык/фреймворк нужен отдельный набор правил, чувствительный к версиям библиотек и стилю кода.

### Подход 2 — RAG-пайплайн

Классическая схема: чанкинг кода → эмбеддинги (`gte-multilingual-base`) → векторный поиск. Затем — гибридный вариант с BM-25-реранкингом поверх.

Результаты:
- **Средняя точность по всем языкам: 85%** — при том что задача плохо прощает ошибки (пропущенный эндпоинт = дыра в документации).
- Поиск медленный.
- Нужные файлы регулярно не попадали в топ выдачи.
- Причины отказов непрозрачны: непонятно, почему модель получила именно эти чанки.

### Подход 3 — Regex + grep

Самый очевидный инструмент оказался самым болезненным в поддержке. Чтобы надёжно найти, например, `@GetMapping` в Java, нужно исключить:

- Javadoc-комментарии (`/** @GetMapping ... */`)
- строковые литералы (`"use @GetMapping for..."`)
- импорты и декларации типов
- произвольные упоминания в тексте на русском и английском

Паттерны усложнялись с lookahead/lookbehind и ограничениями по позиции в строке. Но при малейшем изменении форматирования — аннотация переехала на новую строку, появился дополнительный декоратор — всё ломалось. Сотни микросервисов, ежедневные коммиты, легаси с нестандартной архитектурой. Поддерживать «костыльные регулярки» становилось всё дороже.

## 3. Что такое grep-ast

**Репозиторий:** https://github.com/Aider-AI/grep-ast
**Автор:** Пол Готье (Paul Gauthier) — он же создатель Aider
**Реализация:** Python, поверх tree-sitter и tree-sitter-languages (или tree-sitter-language-pack)
**Текущая версия:** 0.9.0

Идея в одном предложении: ищем текст регулярным выражением как grep, но возвращаем совпадение **не как строку, а как структурный элемент AST** — с полным контекстом: в какой функции, в каком классе, на каком уровне иерархии это находится.

### Что такое AST (краткий ликбез из статьи)

**Parse Tree** (дерево разбора) — точное грамматическое дерево, включающее всё: скобки, запятые, точки с запятой. Следует грамматике языка буквально.

**AST (Abstract Syntax Tree)** — упрощённая семантическая версия: содержит только структурно значимые узлы (операции, переменные, выражения), без избыточных символов. Именно так компилятор/интерпретатор «понимает» код.

grep-ast использует tree-sitter для построения AST, а затем при поиске показывает результат в контексте этого дерева — какие узлы AST стоят выше и ниже совпадения.

### Технические характеристики (CLI)

```
grep-ast [pattern] [filenames...]
```

Альтернативное имя команды: `gast`

| Флаг | Назначение |
|---|---|
| `-i`, `--ignore-case` | Поиск без учёта регистра |
| `--color` / `--no-color` | Управление цветовым выделением (по умолчанию: автодетект TTY) |
| `--encoding ENCODING` | Кодировка файлов (по умолчанию: `utf8`) |
| `-n`, `--line-number` | Показывать номера строк |
| `--no-gitignore` | Игнорировать `.gitignore` |
| `--languages` | Показать таблицу поддерживаемых парсеров |
| `--verbose` | Расширенный вывод (включая AST-узлы) |

- Рекурсивный обход репозитория по умолчанию
- Уважает `.gitignore`
- Поддержка **100+ расширений файлов** через `tree-sitter-language-pack` (с fallback на `tree-sitter-languages`)
- Автоматически пропускает dot-файлы и dot-директории

## 4. Внутренняя механика grep-ast (анализ исходного кода)

Для глубокого понимания того, что именно делает grep-ast, я проанализировал исходный код библиотеки (версия 0.9.0). Это критически важно для оценки статьи, поскольку выявляет существенное расхождение между тем, что описано в статье, и тем, как инструмент работает на самом деле.

### 4.1. Архитектура: три фазы работы

grep-ast работает в три последовательные фазы:

```
Фаза 1: Парсинг        → tree-sitter строит AST, walk_tree() индексирует scopes
Фаза 2: Текстовый поиск → grep() — обычный re.search() по строкам
Фаза 3: Контекстуализация → add_context() расширяет результат до AST-структур
```

#### Фаза 1 — Индексация AST (`walk_tree`)

При инициализации `TreeContext` происходит:

1. Файл парсится tree-sitter: `parser.parse(bytes(code, "utf8"))`
2. Рекурсивный обход AST-дерева (`walk_tree`), для каждого узла:
   - Записывается, какие AST-узлы начинаются на каждой строке (`self.nodes[start_line]`)
   - Для каждой строки фиксируется, в какие скоупы она входит (`self.scopes[i]`)
   - Для многострочных узлов записывается «заголовок» (`self.header[start_line]`)

Результат: **предвычисленная карта скоупов** — для любой строки файла мгновенно известно, внутри каких функций/классов/блоков она находится.

#### Фаза 2 — Текстовый поиск (`grep`)

```python
def grep(self, pat, ignore_case):
    found = set()
    for i, line in enumerate(self.lines):
        if re.search(pat, line, re.IGNORECASE if ignore_case else 0):
            found.add(i)
    return found
```

**Критический факт: поиск — это чистый `re.search()` по каждой строке.** AST на этом этапе не используется. grep-ast ищет ровно так же, как grep — по тексту. Он найдёт `@GetMapping` и в Javadoc, и в строковом литерале, и в импорте.

#### Фаза 3 — Контекстуализация (`add_context`)

Это главная ценность. Для каждой найденной строки:

1. **Parent context** (`add_parent_scopes`): рекурсивно поднимается по дереву скоупов, добавляя заголовки (первые строки) каждого родительского блока — функции, класса, модуля.

2. **Child context** (`add_child_context`):
   - Если блок маленький (< 5 строк) — показывает целиком.
   - Если большой — показывает ~10% дочерних узлов (минимум 5, максимум 25 строк), отсортированных по размеру.

3. **Margin**: всегда показывает первые 3 строки файла (импорты, объявление пакета).

4. **LOI pad**: добавляет ±1 строку вокруг каждого совпадения.

5. **Last line**: показывает последнюю строку файла (закрывающая скобка/`end`).

6. **Close small gaps**: если между показываемыми строками зазор в 1 строку — заполняет его (убирает «⋮» для единичных пропусков).

#### Формат вывода

```
│ контекстная строка (заголовок скоупа)
│ ...
█ строка с совпадением (выделена красным в терминале)
│ контекстная строка
⋮                        ← пропущенные строки
│ закрывающая строка блока
```

- `█` — строка, содержащая совпадение (lines of interest)
- `│` — контекстная строка
- `⋮` — пропущенные строки (elision)

### 4.2. Обнаружение файлов (`enumerate_files`)

Рекурсивный обход директорий через `Path.iterdir()`:
- Пропускает файлы и директории, начинающиеся с `.`
- Для `.gitignore` ищет **ближайший родительский** `.gitignore` файл
- **Ограничение:** обрабатывает только один `.gitignore` — не поддерживает вложенные `.gitignore` в поддиректориях, `.git/info/exclude` или глобальный gitignore
- Файлы с неизвестным расширением или ошибкой декодирования молча пропускаются

### 4.3. Карта поддерживаемых языков

При использовании `tree-sitter-language-pack` (по умолчанию с версии 0.9.0) поддерживается **свыше 100 расширений файлов**, включая:

Все пять языков из статьи поддержаны полностью:
- **Go:** `.go`, `go.mod`, `go.sum`
- **Kotlin:** `.kt`, `.kts`
- **Ruby:** `.rb`
- **Elixir:** `.ex`, `.exs`
- **PHP:** `.php`

Также: Python, JavaScript/TypeScript (включая `.tsx`, `.jsx`), Java, Rust, C/C++, C#, Scala, Swift, Dart, Haskell, OCaml, Erlang, Lua, R, Julia, и десятки других — от Solidity и Cairo до Zig и Odin.

## 5. Конкретные примеры сравнения (из статьи)

### Пример 1 — поиск метода в одном файле

**Задача:** найти метод `check_requirement` в `helpers.py`

**Обычный grep:**
```bash
grep -n "def check_requirement" mini_project/helpers.py
# → mini_project/helpers.py:47:    def check_requirement(self, ...):
```
Получаем номер строки. Чтобы понять: в каком классе находится метод, где его границы, что за docstring — нужно открывать файл вручную.

**grep с контекстом:**
```bash
grep -n -B 3 -A 10 "def check_requirement" mini_project/helpers.py
```
Показывает произвольное количество строк до/после — не знает, где заканчивается метод, легко захватывает код соседней функции, теряет информацию о классе.

**grep-ast:**
```bash
grep-ast "def check_requirement" mini_project/helpers.py
```
Возвращает **весь метод целиком**, внутри правильного класса (`JudgeAgent`), с docstring — без ручного подбора числа строк и без риска потерять структуру.

**Что происходит внутри:** grep-ast находит строку с `def check_requirement`, затем через предвычисленные скоупы определяет, что эта строка находится внутри класса `JudgeAgent`, добавляет заголовок класса, тело метода (через child context), и закрывающую строку.

### Пример 2 — поиск по всему проекту

**Задача:** найти все упоминания `JudgeAgent` в проекте

**Обычный grep:**
```bash
grep -r "JudgeAgent" mini_project/ --include="*.py"
```
Возвращает смешанный список: объявления, импорты, использования — без разбора их отношений. Попутно может найти `NotJudgeAgent`. Для понимания каждого вхождения — снова открывать файл.

**grep-ast:**
```bash
grep-ast "JudgeAgent" mini_project/
```
Группирует результаты по файлу, показывает структурный контекст каждого вхождения: где класс объявлен, где используется, в каком логическом блоке. Десятки ручных переходов между файлами — исчезают.

### Ключевая формула из статьи

> «Grep отвечает на вопрос "где встречается этот текст?", а grep-ast — на вопрос "какие части программы стоят за этим?"»

Для анализа кода и особенно для AI-агентов второй вопрос несравнимо ценнее.

## 6. grep-ast vs ast-grep: важное разграничение

Авторы специально разбирают путаницу — названия похожи, оба используют tree-sitter, оба работают с AST:

| Критерий | **grep-ast** | **ast-grep** |
|---|---|---|
| Язык реализации | Python | Rust |
| Производительность | Умеренная | Очень высокая |
| Основная задача | Поиск с AST-контекстом | Поиск + замена + линтинг |
| Тип паттернов | Regex по тексту | Структурные AST-паттерны |
| Рефакторинг | Нет | Да (интерактивный codemod) |
| Пользовательские правила | Нет | Да (YAML-конфиги) |
| Интерфейсы | CLI | CLI + Node.js API + LSP |

**Вывод команды:** ast-grep избыточен для их задачи — слишком сложный инструмент с функциональностью, которая не нужна. grep-ast решает одну задачу и делает это просто.

### Комментарий к сравнению

Решение команды отказаться от ast-grep стоит рассмотреть критически. ast-grep позволяет искать **структурным паттерном** — например, `@RestController class $NAME { $$$ }` найдёт именно аннотированные контроллеры, не реагируя на `@RestController` в комментариях или строках. Это решает ровно ту проблему с ложными срабатываниями, которую авторы описывают. Аргумент «слишком сложный» — допустим, но стоило бы конкретизировать: сложнее настройка? сложнее интеграция? сложнее паттерны?

## 7. Интеграция в LLM-агент: главная архитектурная идея

Ключевой инсайт статьи — **не в том, что grep-ast лучше grep**, а в том, как его использовать в связке с LLM.

### Классический RAG (что было до)

```
Кодовая база → чанки → эмбеддинги → индекс
                                         ↓
              Запрос → эмбеддинг → топ-K чанков → LLM → ответ
```

Проблемы:
- LLM получает заранее нарезанные куски и вынужден «угадывать» по ним.
- Нет возможности уточнить поиск по ходу рассуждения.
- Причины провала непрозрачны: почему именно эти чанки попали в топ?
- Чанкинг кода — отдельная нетривиальная задача: где разрезать? по функциям? по строкам? с перекрытием?

### grep-ast как tool call (что стало)

```
LLM-агент
  → grep-ast("@RestController") → структурные результаты
  → grep-ast("OrderService")    → уточняем бизнес-логику
  → grep-ast("kafka.*produce")  → находим Kafka-продюсер
  → формируем документацию
```

Агент действует как разработчик: итеративно исследует кодовую базу, уточняет запросы на основе промежуточных результатов, строит понимание от общего к частному.

### Почему это лучше RAG для данной задачи

| Свойство | RAG | grep-ast tool call |
|---|---|---|
| Инициатива | У ретривера (заранее нарезанные чанки) | У агента (сам формирует запросы) |
| Итеративность | Один запрос → один набор чанков | Цепочка запросов, каждый уточняет предыдущий |
| Воспроизводимость | Зависит от эмбеддингов, модели, параметров | Тот же паттерн → тот же результат |
| Инфраструктура | Векторная БД, эмбеддер, индексация | Python-пакет, zero infrastructure |
| Актуальность | Требует переиндексации при изменениях кода | Всегда работает с текущим кодом |
| Стоимость | Хранение эмбеддингов, вычисление при запросе | Только CPU на парсинг |

### Трассируемость как системное свойство

Каждый вызов grep-ast логируется автоматически через tool-calling обёртку:
- какой паттерн использовался,
- какие файлы нашлись,
- куда пошло исследование дальше.

Цепочка рассуждений агента становится **полностью читаемой**:
```
1. grep-ast "@RestController"  → нашли контроллеры заказов
2. grep-ast "createOrder"      → нашли метод создания заказа
3. grep-ast "OrderService"     → нашли слой сервисов
4. grep-ast "KafkaTemplate"    → нашли Kafka-продюсер в сервисе
```

Это принципиальное отличие от RAG: там чёрный ящик на этапе ретривала. Здесь — детерминированный, логируемый, воспроизводимый процесс.

### Контекст: как это делают современные AI-инструменты для кода

Паттерн «детерминированный поиск как tool call» — не изобретение ecom.tech. Он используется в промышленных AI-инструментах:

- **Aider** (автор grep-ast — Пол Готье) использует grep-ast как часть своего «repo map» — карты репозитория, которую агент использует для навигации.
- **Claude Code** использует `Grep`, `Glob`, `Read` как tool calls — агент итеративно ищет файлы, читает их, формирует понимание.
- **Cursor** комбинирует RAG-индексацию с прямыми инструментами поиска, доступными агенту.

Подход ecom.tech вписывается в этот тренд, но фокусируется на **узкой задаче** (поиск API-эндпоинтов), а не на универсальной навигации по коду.

## 8. Критический анализ

### 8.1. Сильные стороны статьи

**Честная эволюция решений.** Авторы не начинают сразу с победного результата — показывают три тупика, с конкретными проблемами каждого. Это ценнее, чем типичная «мы взяли X и всё стало хорошо».

**Правильный уровень абстракции.** Статья не уходит в детали реализации tree-sitter и не остаётся на уровне «мы сделали поиск». Объяснение AST vs Parse Tree — ровно столько, сколько нужно читателю.

**Нетривиальный архитектурный паттерн.** grep-ast как tool call для итеративной навигации — это не очевидный приём. Большинство команд либо делают RAG, либо дают агенту весь файл целиком. Промежуточный вариант с детерминированным структурным поиском практически не описан в публичных материалах.

**Разграничение с ast-grep.** Путаница реальная, раздел полезный.

**Акцент на трассируемость.** Это системное свойство, которое часто игнорируется при выборе инструментов для AI-агентов. Авторы правильно выводят его на первый план.

### 8.2. Критические проблемы

#### Проблема 1: grep-ast НЕ решает проблему ложных срабатываний

**Это ключевое расхождение между описанием в статье и реальностью.**

Статья описывает проблему regex-подхода: `@GetMapping` в Javadoc, `// GetUser` в комментариях — и подразумевает, что grep-ast эту проблему решает благодаря «пониманию синтаксической структуры кода».

Анализ исходного кода показывает обратное. Метод `grep()` в `TreeContext`:

```python
def grep(self, pat, ignore_case):
    for i, line in enumerate(self.lines):
        if re.search(pat, line, ...):
            found.add(i)
    return found
```

Это **чистый `re.search()` по каждой строке** — идентично grep. AST используется **только для контекстуализации** (показать найденное в структуре), но **не для фильтрации** (отличить код от комментариев/строк).

Если `@GetMapping` написано в Javadoc — grep-ast его найдёт и покажет в контексте (внутри какого класса/метода этот Javadoc). Если `// GetUser` — в комментарии — найдёт и покажет с контекстом. Ложные срабатывания **не устраняются**.

Что grep-ast реально делает лучше grep: показывает контекст осмысленно. LLM, получив контекст, может сама отличить комментарий от кода. Но это свойство **модели**, а не инструмента — и об этом в статье не сказано.

#### Проблема 2: асимметрия измерений

RAG-пайплайн получил конкретную цифру — 85%. Для grep-ast цифры нет. Декларируется, что стало значительно лучше, но без числа это звучит как маркетинг. Хотя бы порядок: 95%? 99%? Это существенно.

Более того, не уточняется, что именно измерялось. «Средняя точность» — precision? recall? F1? По каким языкам средняя — арифметическая или взвешенная? Один язык с 60% может прятаться за четырьмя с 91%.

#### Проблема 3: ложные отрицания при динамической регистрации

Статья описывает проблемы ложных срабатываний (false positives), но молчит о ложных отрицаниях (false negatives). Grep-ast ищет по тексту — он найдёт только то, что написано явно. Что происходит с:

- **Rails `resources :orders`** — маршруты генерируются автоматически, в коде нет `GET /orders`. Грепать по `resources` можно, но нужно знать, что `resources :orders` порождает 7 маршрутов (index, show, new, create, edit, update, destroy). Это семантика фреймворка, которую ни grep, ни grep-ast не понимают.
- **Elixir-макросами**, которые регистрируют консьюмеры Kafka во время компиляции. Макрос `use MyApp.KafkaConsumer, topic: "orders"` в AST — просто вызов функции `use`.
- **Golang-кодом**, где роуты регистрируются в цикле из конфигурационного файла:
  ```go
  for _, route := range config.Routes {
      router.Handle(route.Method, route.Path, route.Handler)
  }
  ```
  Ни grep, ни grep-ast не свяжут `router.Handle` с конкретными эндпоинтами — для этого нужно читать конфигурацию.

Это реальные паттерны в production-коде, и для них текстовый поиск не поможет.

#### Проблема 4: управление паттернами — белое пятно

В разделе о задаче авторы перечисляют ~15 различных паттернов для пяти языков. Как они хранятся? Как обновляются при смене версии фреймворка? Кто отвечает за их актуальность? Это ключевая инженерная проблема решения, и она не раскрыта.

Вопросы, которые остаются без ответа:
- Паттерны зашиты в промпт агента или в конфигурацию?
- Есть ли fallback, когда паттерн не нашёл ничего?
- Как обрабатывается ситуация, когда один проект использует нестандартный фреймворк?
- Тестируются ли паттерны на regression при обновлении?

#### Проблема 5: промпт агента не показан

Сказано, что LLM делает итеративные запросы к grep-ast. Но: как агент получает инструкцию делать именно итеративный поиск, а не один запрос? Каков промпт? Как обрабатывается случай, когда первый запрос не нашёл ничего полезного?

Это центральная часть системы. Без неё описание архитектуры — декларация намерений, а не воспроизводимое решение. Конкретные вопросы:
- Какой шаблон tool call описания передаётся модели?
- Есть ли few-shot примеры итеративного поиска в промпте?
- Как модель понимает, что поиск нужно продолжить (а не остановиться на первом результате)?
- Какая модель используется? (GPT-4, Claude, что-то open-source?)
- Сколько итераций в среднем нужно на один микросервис?

#### Проблема 6: производительность на масштабе

«Сотни микросервисов» — это может быть 100 и может быть 500. Grep-ast написан на Python поверх tree-sitter.

Конкретные bottleneck'и, видимые в коде:
- **Нет параллелизма:** файлы обрабатываются последовательно в цикле `for fname in enumerate_files(...)`.
- **Нет кеширования:** AST строится заново при каждом вызове. Если агент делает 5 запросов к одному файлу — файл парсится 5 раз.
- **Python GIL:** даже если добавить threading, tree-sitter FFI-вызовы не освободят GIL.
- **Полная загрузка в память:** `file.read()` без streaming — для больших файлов это может быть проблемой.

Авторы отмечают, что RAG был «медленным» — но RAG медленный из-за эмбеддингов и поиска по индексу. grep-ast медленный из-за парсинга каждого файла при каждом запросе. Что быстрее в их сценарии — неизвестно.

#### Проблема 7: ограничение gitignore

Код обрабатывает только **один `.gitignore`** — ближайший родительский. В реальном монорепозитории или в проектах с несколькими микросервисами в одной директории часто есть вложенные `.gitignore`. Код grep-ast их проигнорирует, что может привести к обработке файлов, которые не должны попадать в поиск (vendor/, node_modules/ в поддиректориях).

#### Проблема 8: отказ от ast-grep недостаточно обоснован

Авторы написали, что ast-grep — «слишком сложный инструмент с избыточным функционалом». Но ast-grep решает ровно ту проблему, которая у авторов осталась нерешённой: **структурный поиск, который отличает код от комментариев и строковых литералов**.

Пример: в ast-grep можно написать паттерн `@RestController class $C { $$$ }`, и он найдёт именно классы с этой аннотацией, игнорируя упоминания в Javadoc. В grep-ast это невозможно — он не фильтрует по типу AST-узла.

Если «сложность» — это порог входа для написания YAML-правил, это справедливый аргумент. Но стоило бы его конкретизировать: сколько времени заняло бы написание ast-grep паттернов для 5 языков vs. поддержка regex-паттернов + обработка false positives моделью?

## 9. Более широкий контекст

### Паттерн «детерминированный инструмент как tool call» недооценён

Большинство публичных материалов об AI-агентах для кода предлагают два полюса:
- **RAG**: семантический поиск, чёрный ящик, трудно отлаживать.
- **Full-context**: отдать агенту весь репозиторий (работает только для маленьких проектов).

grep-ast как tool call — это третий путь: **детерминированный, структурный, итеративный**. Агент сам управляет навигацией, каждый шаг прозрачен, результаты воспроизводимы.

### Когда AST-поиск лучше эмбеддингов

| Сценарий | AST-поиск | Эмбеддинги |
|---|---|---|
| Найти все `@RestController` в проекте | ✅ Точно, быстро | ⚠️ Избыточно сложно |
| Найти «код, связанный с авторизацией» | ❌ Нужны явные паттерны | ✅ Семантика помогает |
| Найти реализации интерфейса | ✅ Структурный поиск | ⚠️ Работает, но хуже |
| Найти «похожую логику» | ❌ Текстовый поиск не справится | ✅ Эмбеддинги для этого |
| Найти динамически генерируемые маршруты | ❌ Не найдёт | ⚠️ Может, если обучена |
| Отличить код от комментариев | ❌ grep-ast не умеет | ⚠️ Контекстно |

**Вывод:** AST-поиск — не замена эмбеддингам, а альтернатива для структурно определённых целей. Но и AST-поиск в grep-ast ограничен — для настоящего структурного поиска (с фильтрацией по типу узла) нужен ast-grep.

### Гибридный подход: что авторы могли бы рассмотреть

Оптимальная архитектура для задачи ecom.tech, возможно, выглядит так:

```
1. grep-ast / ast-grep → находит структурные паттерны (эндпоинты, продюсеры)
2. Для неоднозначных случаев → RAG с эмбеддингами как fallback
3. LLM-агент → итеративно использует оба инструмента
4. Пост-обработка → LLM фильтрует false positives, дополняет false negatives
```

Статья описывает grep-ast как **замену** RAG. Более продуктивный подход — **комбинация**: AST-поиск для структурно определённых целей, эмбеддинги для семантически размытых.

## 10. Контекст: grep-ast в экосистеме Aider

grep-ast — не изолированная библиотека, а часть инструментария Aider. Пол Готье создал её для конкретной задачи: дать Aider возможность быстро строить «repo map» — сжатую карту репозитория, показывающую структуру кода без полного содержания файлов.

В Aider grep-ast используется для:
- Определения, какие файлы релевантны текущей задаче
- Построения контекста, который помещается в промпт
- Показа пользователю, какие части кода Aider «видит»

Это объясняет дизайн-решения grep-ast:
- **Почему текстовый поиск, а не структурный:** Aider не знает заранее, что искать — паттерн формируется LLM или пользователем в свободной форме.
- **Почему нет фильтрации по типу узла:** для Aider важнее показать контекст (чтобы LLM сама поняла), чем отфильтровать (и рискнуть потерять релевантное).
- **Почему простой CLI:** Aider вызывает grep-ast программно, ему не нужен сложный API.

Команда ecom.tech использует grep-ast в аналогичном контексте (tool call для LLM-агента), что подтверждает его пригодность для этой роли. Но ограничения, унаследованные от дизайна Aider, остаются.

## 11. Двухстадийный пайплайн: grep-ast → ast-grep

### 11.1. Постановка вопроса

Из анализа исходного кода в разделе 8 следует ключевое ограничение grep-ast: он ищет регулярным выражением по тексту, не фильтруя по типу AST-узла. Комментарий с `@GetMapping` — ложное срабатывание, которое grep-ast найдёт так же, как и настоящую аннотацию.

В то же время ast-grep умеет делать именно то, чего не хватает grep-ast: **структурный поиск**, при котором паттерн привязан к типу AST-узла. Вопрос: можно ли использовать эти инструменты последовательно — grep-ast как быстрое сито первой стадии, ast-grep как точный фильтр второй?

Ответ: **да, это не просто возможно — это естественное распределение ролей.**

---

### 11.2. Что каждый инструмент делает хорошо

| Задача | grep-ast | ast-grep |
|---|---|---|
| Быстрое обнаружение файлов с нужным текстом | ✅ | ✅ |
| Показать совпадение в структурном контексте (для LLM) | ✅ | ❌ (JSON/plain output) |
| Различить код, комментарий, строковый литерал | ❌ | ✅ |
| Захватить переменные части паттерна (`$PATH`, `$HANDLER`) | ❌ | ✅ |
| Простота написания паттернов | ✅ (regex) | ⚠️ (своя DSL) |
| Программный Python API | ✅ (`TreeContext`) | ⚠️ (через CLI/Node.js) |
| Скорость на большом объёме | ⚠️ (Python) | ✅ (Rust) |
| Поддержка `.gitignore` | ✅ (частичная) | ✅ |

Инструменты дополняют друг друга: у каждого есть сильная сторона там, где у другого слабость.

---

### 11.3. Архитектура двухстадийного пайплайна

```
СТАДИЯ 1: grep-ast (широкий текстовый поиск)
─────────────────────────────────────────────
Вся кодовая база (1000 файлов)
  → паттерн: "RestController|RequestMapping"
  → результат: 40 файлов + структурный контекст для LLM
  → выход: список файлов-кандидатов

СТАДИЯ 2: ast-grep (структурная валидация)
──────────────────────────────────────────
40 файлов-кандидатов
  → паттерн: "@RestController\nclass $CLASS { $$$ }"
  → результат: 38 файлов (2 ложных срабатывания отброшены)
  → выход: точные координаты совпадений (файл, строка, захваченные переменные)

СТАДИЯ 3: LLM
──────────────────────────────────────────
Берёт grep-ast контекст (для читаемости) +
     ast-grep координаты (для точности)
  → формирует документацию
```

Ключевой момент: **стадии работают на разных объёмах данных**. grep-ast обрабатывает весь репозиторий, но делает это быстро (regex). ast-grep получает только файлы-кандидаты — их мало, и Rust-движок обрабатывает их мгновенно.

### 11.4. Конкретные паттерны для задачи ecom.tech

Покажем, как выглядела бы пара паттернов для каждого языка:

#### Kotlin — Spring @RestController

**Стадия 1 (grep-ast, regex):**
```
@RestController|@RequestMapping|@GetMapping|@PostMapping
```

**Стадия 2 (ast-grep, YAML-правило):**
```yaml
id: kotlin-rest-controller
language: Kotlin
rule:
  kind: class_declaration
  has:
    pattern: "@RestController"
```
Это правило найдёт именно `class_declaration` с аннотацией `@RestController` — и не найдёт `@RestController` в Javadoc или строковом литерале.

#### Golang — Gin/Echo маршруты

**Стадия 1 (grep-ast, regex):**
```
\.GET\(|\.POST\(|\.PUT\(|\.DELETE\(|HandleFunc
```

**Стадия 2 (ast-grep, паттерн):**
```yaml
id: go-gin-route
language: Go
rule:
  any:
    - pattern: $ROUTER.GET($PATH, $HANDLER)
    - pattern: $ROUTER.POST($PATH, $HANDLER)
    - pattern: http.HandleFunc($PATH, $HANDLER)
```
Захватывает переменные `$PATH` и `$HANDLER` — grep-ast этого не умеет вовсе.

#### PHP — Laravel routes/api.php

**Стадия 1 (grep-ast, regex):**
```
Route::get|Route::post|Route::apiResource
```

**Стадия 2 (ast-grep, паттерн):**
```yaml
id: php-laravel-route
language: PHP
rule:
  any:
    - pattern: Route::get($PATH, $HANDLER)
    - pattern: Route::post($PATH, $HANDLER)
    - pattern: Route::apiResource($PATH, $CONTROLLER)
```

#### Elixir — Phoenix Router

**Стадия 1 (grep-ast, regex):**
```
get "|post "|resources "
```

**Стадия 2 (ast-grep, паттерн):**
```yaml
id: elixir-phoenix-route
language: Elixir
rule:
  pattern: get $PATH, $CONTROLLER, $ACTION
```

**Важное замечание:** ast-grep поддерживает Elixir через tree-sitter, но макросы Phoenix — это `call`-узлы, синтаксически неотличимые от обычных функций. Для Elixir структурный паттерн уберёт ложные срабатывания из комментариев, но не решит проблему макросов, генерирующих маршруты.

#### Kafka — Kotlin KafkaTemplate

**Стадия 1 (grep-ast, regex):**
```
kafkaTemplate|KafkaTemplate|\.send\(
```

**Стадия 2 (ast-grep, паттерн):**
```yaml
id: kotlin-kafka-send
language: Kotlin
rule:
  pattern: $TEMPLATE.send($TOPIC, $$$)
  inside:
    kind: function_declaration
```
Дополнительное ограничение `inside: function_declaration` — находим вызов `.send()` только внутри тела функции, не в строках документации.

### 11.5. Плюсы двухстадийного пайплайна

**1. Естественное разделение ответственности**
grep-ast делает то, что умеет лучше всего — быстро сужает поле поиска. ast-grep делает то, что умеет лучше всего — структурно валидирует. Ни один из инструментов не нагружается не своей задачей.

**2. Устранение false positives без потери false negatives**
Стадия 1 намеренно широкая — она не должна пропустить ничего. Стадия 2 отбрасывает лишнее. Риск: если стадия 1 что-то пропустила (паттерн слишком узкий) — стадия 2 это не вернёт. Но контроль над этим риском простой: паттерны стадии 1 пишутся максимально широкими.

**3. Захват переменных — принципиально новая возможность**
ast-grep возвращает не просто «файл + строка», а структурированный результат:
```json
{
  "file": "controllers/orders.go",
  "line": 42,
  "vars": {
    "PATH":    "\"/api/v1/orders\"",
    "HANDLER": "HandleCreateOrder"
  }
}
```
Это напрямую используется для генерации документации — без необходимости просить LLM «извлеки путь и хендлер из этого кода».

**4. Производительность масштабируется**
На репозитории из 10 000 файлов:
- grep-ast (стадия 1): обрабатывает все 10 000 файлов, находит 200 кандидатов
- ast-grep (стадия 2): обрабатывает 200 файлов на Rust — единицы миллисекунд

Итого: почти вся вычислительная нагрузка — на стадии 1 (Python), но grep-ast на чистом regex работает быстро.

**5. Снижение нагрузки на LLM**
Без стадии 2: LLM получает вывод grep-ast с false positives и тратит контекстное окно на их анализ и отброс.
Со стадией 2: LLM получает уже отфильтрованный список + структурированные переменные. Меньше токенов, меньше вероятность галлюцинаций.

**6. Трассируемость сохраняется**
Обе стадии детерминированы и логируемы. Цепочка стадий ещё информативнее:
```
grep-ast "RestController" → 40 кандидатов
ast-grep "@RestController class" → 38 подтверждённых (2 отброшено из комментариев)
```

### 11.6. Минусы и риски

**1. Удвоение поддержки паттернов**
Сейчас команда поддерживает N regex-паттернов. При двухстадийном подходе — N regex + N структурных YAML. Это реальная операционная нагрузка. Противоядие: автотесты на эталонных репозиториях; структурные паттерны меняются реже (они стабильнее версий фреймворка).

**2. Порог входа в ast-grep DSL**
Авторы статьи назвали ast-grep «слишком сложным». Оценим честно:

| Сложность | Оценка |
|---|---|
| Простой pattern (`$ROUTER.GET($PATH, $HANDLER)`) | Низкая — интуитивно |
| Правила с `kind`, `inside`, `has` | Средняя — нужно знать AST-типы языка |
| Комбинации `all`/`any`/`not` | Высокая — приближается к Datalog |
| Дебаггинг при неработающем правиле | Высокая — нет хороших сообщений об ошибках |

Для задачи ecom.tech достаточен уровень «Простой + Средний». Это несколько часов изучения, не недели.

**3. Каскад ошибок стадии 1**
Если grep-ast паттерн пропустил файл (false negative) — ast-grep его никогда не увидит. Нужна политика: паттерны стадии 1 **намеренно широкие**, паттерны стадии 2 **намеренно узкие**.

**4. Несовместимость языкового покрытия**
Оба используют tree-sitter, но наборы поддерживаемых языков могут различаться. Для пяти языков задачи — Golang, Kotlin, Ruby, Elixir, PHP — оба инструмента поддерживают все. Но в будущем при добавлении нового языка нужно проверять оба.

**5. Вывод ast-grep менее LLM-friendly**
grep-ast возвращает человекочитаемый текст с визуальной иерархией (`│`, `█`, `⋮`). ast-grep возвращает JSON или plain text без структурного контекста. LLM работает лучше с grep-ast выводом — он содержит больший контекст. Решение: передавать LLM grep-ast вывод (для контекста), ast-grep вывод (для точных координат).

**6. Elixir и Ruby — структурные паттерны не решают макро-проблему**
Для языков с метапрограммированием ast-grep устраняет ложные срабатывания из комментариев/строк, но не помогает с эндпоинтами, генерируемыми макросами. Для Ruby `resources :orders` — ast-grep найдёт вызов `resources`, но не знает, что он порождает 7 маршрутов. Это фундаментальное ограничение обоих инструментов.

### 11.7. Сравнение архитектурных вариантов

| Вариант | Точность | Сложность | Производительность | Трассируемость |
|---|---|---|---|---|
| grep-ast alone | ⚠️ False pos. из комментариев | Низкая | Средняя (Python) | ✅ |
| ast-grep alone | ✅ Структурная | Высокая | ✅ (Rust) | ✅ |
| grep-ast → ast-grep | ✅ Структурная | Средняя | ✅ | ✅ |
| RAG | ⚠️ Семантическая (85%) | Высокая (инфраструктура) | ⚠️ Медленная | ❌ |
| grep-ast + RAG fallback | ✅ + покрытие семантики | Высокая | ⚠️ | ✅ / ❌ |

**Оптимальный выбор для задачи ecom.tech:** вариант `grep-ast → ast-grep`, с RAG как fallback для случаев, где оба инструмента возвращают пустой результат (динамически генерируемые маршруты, нестандартные фреймворки).

### 11.8. Практическая схема tool calls для LLM-агента

При двухстадийном подходе агент получает три инструмента вместо одного:

```python
tools = [
    {
        "name": "grep_ast",
        "description": (
            "Широкий текстовый поиск по кодовой базе с AST-контекстом. "
            "Используй для первичного обнаружения файлов и получения "
            "структурного контекста. Возвращает файлы и их структуру. "
            "Паттерн: regex."
        ),
        "parameters": {"pattern": "str", "path": "str"}
    },
    {
        "name": "ast_grep",
        "description": (
            "Структурный поиск по AST. Использует после grep_ast для "
            "точной валидации. Отличает код от комментариев и строк. "
            "Захватывает переменные части ($PATH, $HANDLER). "
            "Паттерн: ast-grep DSL."
        ),
        "parameters": {"pattern": "str", "language": "str", "files": "list[str]"}
    },
    {
        "name": "read_file",
        "description": "Читает конкретный файл целиком если нужен полный контекст.",
        "parameters": {"path": "str"}
    }
]
```

Агент сам решает, когда какой инструмент использовать. Типичный сценарий:

```
1. grep_ast("@RestController", path="services/orders/")
   → нашли 12 файлов, показан структурный контекст

2. ast_grep("@RestController\nclass $C { $$$ }", language="kotlin",
            files=["OrdersController.kt", "BaseController.kt", ...])
   → 11 подтверждённых контроллеров, 1 ложное срабатывание отброшено
   → захвачены переменные: CLASS="OrdersController", ...

3. grep_ast("KafkaTemplate", path="services/orders/")
   → 3 файла с Kafka-продюсерами

4. ast_grep("$T.send($TOPIC, $$$)", language="kotlin",
            files=["OrdersService.kt", ...])
   → 5 точных точек вызова send() с захваченными топиками
```

Итог: **LLM не тратит контекстное окно на разбор false positives.** Она получает структурированный, проверенный результат и сразу генерирует документацию.

## 12. Итоговые выводы для практиков

**1. Перед RAG задайте вопрос: структурно ли определена цель?**
Если ищете классы, декораторы, специфические вызовы — AST-поиск точнее, быстрее и прозрачнее. Эмбеддинги нужны для семантически размытых запросов.

**2. grep-ast — хороший tool call для LLM-агентов в кодовых базах.**
Итеративная навигация надёжнее одноразового RAG-ретривала. Агент с grep-ast думает как разработчик, а не как поисковик.

**3. Трассируемость — системное свойство, а не приятный бонус.**
Детерминированный логируемый инструмент делает поведение агента понятным, отлаживаемым и объяснимым заказчику.

**4. Понимайте, что именно делает ваш инструмент.**
grep-ast показывает контекст, но не фильтрует по типу AST-узла. Фильтрация ложных срабатываний ложится на LLM, а не на инструмент. Это важно для проектирования промпта.

**5. Библиотека узкоспециализированная — это и плюс, и ограничение.**
grep-ast не пытается быть Swiss Army knife. Одна задача, простой интерфейс, предсказуемое поведение. Но для настоящего структурного поиска (отличить код от комментариев) нужны другие инструменты.

**6. Измеряйте оба конца сравнения.**
85% для baseline — хорошо. Но без цифры для нового решения это не сравнение, а анекдот.

**7. Учитывайте ложные отрицания.**
Любой инструмент поиска по тексту (включая grep-ast) пропустит динамически генерируемые структуры. Знайте границы своего инструмента и имейте fallback.

**8. Рассмотрите гибридную архитектуру.**
AST-поиск для структурных целей + RAG для семантических + LLM для фильтрации и синтеза — вероятно, лучше любого одного подхода.

**9. grep-ast → ast-grep — естественный двухстадийный пайплайн.**
grep-ast как широкое сито первой стадии (находит файлы, даёт LLM-friendly контекст), ast-grep как структурный фильтр второй стадии (устраняет false positives, захватывает переменные части паттерна). Инструменты не конкурируют — они покрывают разные слои задачи.

**10. Паттерны стадии 1 — широкие, стадии 2 — узкие.**
Главный риск двухстадийной схемы: false negative на стадии 1 каскадирует. Противоядие: regex стадии 1 намеренно широкий (лучше лишние кандидаты, чем пропущенные), YAML-правило стадии 2 — точное.

**11. ast-grep захватывает переменные — это снижает нагрузку на LLM.**
Вместо «LLM, извлеки путь и хендлер из этого кода» вы получаете `{"PATH": "/api/orders", "HANDLER": "HandleCreate"}` напрямую из инструмента. Меньше токенов, меньше ошибок экстракции.

**12. Для Elixir и Ruby метапрограммирование — за пределами любого текстового поиска.**
Ни grep-ast, ни ast-grep не решат проблему маршрутов, генерируемых макросами. Здесь нужна или статическая интерпретация кода, или комбинация с анализом routes-файлов / рефлексией во время выполнения.
